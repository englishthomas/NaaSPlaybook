1 Introduction

The Mobile Core Network is the set of elements responsible for signaling handling, connection to other networks such as the Internet, and providing mobility to the Radio Access Network (RAN). Within the signaling procedures, it ensures the proper identification of the user, it enables security mechanisms for the connection and provides a set of customizations tailored for the services provided by the network. It also concentrates the data traffic from all the users to offer interconnection to other networks (e.g. internet) while keeping the sessions isolated from user to user (security and privacy).

The Mobile Core Architecture Module aims to enable the NaaS Operator to perform the overall definition of the elements, interfaces, interworking and interconnection mechanisms of the Evolved Packet Core (EPC) in order to support the desired LTE services.

This module provides an overview of the EPC fundamentals addressing the scope and functions of the elements within the mobile architecture and the additional nodes needed to provide extended capabilities such as interconnection to other 3GPP networks or voice services. It provides guidelines for the NaaS operator to build the topology that best suits its capabilities and constraints, considering management and operational aspects, resilience, and performance.

Considering the evolution of the telco architecture, the module presents the NFV framework and its direct relation to the mobile core. The module leads the NaaS operator through a series of guidelines to generate the NFV architecture that works best with the defined EPC logical architecture.

The output of the module is the Mobile Core Reference Architecture, which includes the EPC topology, interworking and interconnection solutions, and finally, the requirements for a virtualized implementation.


1.1 Module Objectives

This module will enable the NaaS Operator to define the reference architecture of the mobile LTE core. The specific objectives of this module are to:

  1. Provide an information base and fundamentals of the architectural elements of the LTE core.
  2. Guide the NaaS operator to define the EPC Architecture and the functional and operational requirements to provide an LTE Service.
  3. Provide instruction to determine the telco cloud architecture to support the LTE core network elements.
  4. Provide guidelines to develop a formal Core Architecture Specification.

2 LTE Core Network Architecture

In this section, the fundamentals of Mobile Core architecture and directives to select core elements for the NaaS Operator architecture are covered, including policy, charging and control/user plane separation (CUPS).

The decision of whether to deploy a Mobile Core and which type of deployment will depend on the NaaS Operator strategy regarding both the services to be offered and the target customers. The variety of customers can fall into the following categories:

  • Mobile Network Operator (MNOs): Operators that own their Mobile Core and Radio Access Network; however, they are potential customers since they do not have RAN coverage in the area where the NaaS Operator will be providing service.
  • Full Mobile Virtual Network Operators (MVNOs): Operators that own a Mobile Core Network but no Radio Access Network. The Mobile Core Network of a full MVNO consist of the complete set of elements required to provide the services offered to their subscribers.
  • Light MVNOs: This type of MVNO only owns a part of the Mobile Core Network, usually consisting of the subscriber database (HSS), the policy and charging rules function (PCRF) for differentiated, and in some cases, the Packet Data Network (PDN) Gateway (PGW). Details of these nodes are explained in section 2.1.
  • Brand Sellers: Operators that do not own any of the infrastructure of the EPC. They are pure virtual operators and their services are carried by another operators RAN and Core network.

As explained above, each type of customer implements a certain part (or none) of the EPC. Thus, to provide the desired services, NaaS Operators must fill the blanks in the network, i.e. implement the necessary core elements to make the service work when connecting to the customers infrastructure. Based on this, Table 1 provides guidance for NaaS Operators to identify at a high level the type of Mobile Core implementation that should be considered for their Network.

2.1 Evolved Packet Core Components

The Evolved Packet Core (EPC) has been designed as a flat all-IP architecture supporting only packet-based services. EPC aims at providing seamless IP connectivity between user equipment (UE) and correspondent applications. The basic EPC architecture and the interaction with the Evolved Universal Terrestrial Radio Access Network (E-UTRAN) are shown in Figure 3.

Figure 3 Evolved Packet Core basic architecture


The elements of the Evolved Packet Core (EPC) are:

  • MME (Mobility Management Entity)
  • S-GW (Serving Gateway)
  • P-GW (Packet Data Network Gateway)
  • HSS (Home Subscriber Server)
  • PCC (Policy and Charging Control) Elements

With the evolution of the Network Function Virtualization (NFV) technology (see Section 5), EPC solutions can be found as a core in a box deployment with the standard EPC components HSS, MME, PGW, SGW and PCRF in one virtualized bundle. Some of the benefits that a virtualized core brings compared to the legacy offer are:

  • Fast deployment: In recent polls, operators were able to turn up a virtualized network much more quickly (less than 6 months) than with traditional networks (15 months on average).
  • Saving costs: Studies have shown that virtualizing the EPC can save up to 60% on CapEx and 60% on OpEx.
  • Load balancing: A virtual EPC platform can include a load-balancing function, which in the hardware world was sold as a separate function with a separate box.
  • Orchestration: Virtual EPCs can be set up using software-defined networking techniques, including orchestration and service chaining.

In the following subsections, a review of the EPC elements and their functions is provided, assuming a virtualized implementation of the core, hence they are kept at high-level (3GPP Specifications Guide can be reviewed for further detail). Afterwards, section 2.5 provides straightforward guidelines for NaaS Operators to define the elements to be implemented as part of their Mobile Core network.

2.1.1 Mobility Management Entity (MME)

Control plane entity within the EPS (Evolved Packet System). The MME is a mandatory element for the NaaS Operator whenever mobile core infrastructure is being deployed in order to have control over the RAN access.

Functions:

  • Mobility Management: The MME deals with the control plane. It handles the signaling related to mobility and security for E-UTRAN access. It handles PDN GW and Serving GW selection.
  • Non-Access Stratum (NAS) signaling and security: The MME is responsible for the tracking and the paging of UE in idle-mode.

2.1.2 Serving Gateway (SGW)

The gateway that terminates the interface towards the access network. This element is always required by the NaaS operator, to have control over the session, route traffic to the Internet or the transport nodes, and also to perform roaming.

Functions:

  • Packet routing and forwarding: The S-GW is the point of interconnection between the radio-side and the EPC. This gateway serves the UE by routing the incoming and outgoing IP packets to transport the IP data traffic between the User Equipment (UE) and the external networks.
  • Local Mobility Anchor point for inter-eNodeB handover.

2.1.3 Packet Data Network Gateway (PDN GW)

The gateway that terminates the SGi interface towards the Packet Data Network (PDN). It provides the NaaS operator connectivity to Packet Data Networks like the Internet, Intranet and IP Multimedia Subsystem (IMS), supporting both 3GPP (3G and 4G) and non-3GPP (i.e. Wi-Fi) access networks. It also assigns IP addresses to endpoints and manages packet routing, filtering and forwarding.

Functions:

  • Per-user based packet filtering (e.g. deep packet inspection).
  • UE IP address allocation.
  • Uplink (UL) and Downlink (DL) service level charging, gating control, rate enforcement as defined in TS 23.203.
  • User plane anchor for mobility between 3GPP access and non-3GPP access.

2.2 Policy and Charging Control (PCC) Elements

The NaaS operator sells network capacity and services to its clients. The elements of the PCC provide access, resource, and Quality-of-Service (QoS) control to differentiate these services. Policy control within the Policy and Charging Rules Function (PCRF) and Policy and Charging Enforcement Function (PCEF) ensure that appropriate amounts of bandwidth are dynamically allocated to each service in real time, thus making the most efficient use of network resources and maximizing profits.

Prior to launching a new service, operators need to test and validate their policy rules within the PCRF and PCEF to ensure the services are delivered with integrity and to ensure that there is sufficient capacity to provide the requested services. A NaaS operator may implement a multitude of charging rules for each service it provides.

The elements of the PCC architecture are shown in Figure 4 and further described in the subsections below. Depending on the scope and target customers, the NaaS operator might only deploy some of them (see section 2.5 for EPC deployment options).

Figure 4 PCC Elements and related interfaces.


2.2.1 Policy and Charging Rules Function (PCRF)

Decision point for policy and charging control of service data flows/applications and IP bearer resources. This function selects and provides the applicable policy and charging control decision and sends it to the PCEF. This is a mandatory element whenever the NaaS operator is implementing the full EPC deployment.

Functions:

  • Encompasses policy control decision and flow-based charging control functionalities.
  • Provides network control regarding service detection and charging towards the PCEF.
  • Decides if application traffic detection is to be applied.
  • Decides how certain service/application traffic shall be treated in the PCEF.

2.2.2 Policy and Charging Enforcement Function (PCEF)

Enforcement point for policy and charging control of IP bearer resources. Usually located in the PDN-GW. It allows the NaaS operator to perform the QoS and gate controls by each flow according to the policy settings. This is a mandatory element whenever the NaaS operator is implementing the full EPC deployment.

Functions:

  • Encompasses service data flow detection, policy enforcement and flow-based charging functionalities.
  • Enforces the Policy Control as indicated by the PCRF in two different ways:
    • Gate enforcement. The PCEF shall allow a service data flow subject to policy control, to pass through the PCEF if and only if the corresponding gate is open;
    • QoS enforcement: The PCEF shall enforce the authorized QoS of a service data flow according to the active PCC rule and PCEF controls the QoS that is provided to a combined set of service data flows (bearer QoS enforcement).
  • Enforces the charging control: For a service data flow (defined by an active PCC rule) that is subject to charging control, the PCEF shall allow the service data flow to pass through the PCEF if and only if there is a corresponding active PCC rule.

2.2.3 Home Subscriber Server (HSS)

Entity containing the subscription-related information to support the network entities handling calls/sessions; it also holds user location information. If the NaaS operator wants to handle end users, this is a must-have network element.

The user-related information that HSS holds is:

  • User Identification, Numbering and addressing information.
  • User Security information: Network access control information for authentication and authorization.
  • User profile information.

Functions:

  • Home Location Register/Authentication Center (HLR/AUC) functionality. It enables subscriber access and stores an identity key for each mobile subscriber registered with the associated HLR.
  • Mobility Management Support
  • Call and/or session establishment support
  • Access authorization
  • Service authorization and provisioning

2.2.4 Online Charging System (OCS)

This subsystem provides credit management and grants credit to the PCEF based on time, traffic volume or chargeable events. If the NaaS operator charging is based on pre-paid schemas, this is the billing platform to be used.

2.2.5 Off-line Charging System (OFCS)

This subsystem receives events from the PCEF and generates charging data records (CDRs) for the billing system. For the NaaS operator it handles the roaming and interconnection billing based on CDRs.

2.2.6 Subscription Profile Repository (SPR)

The SPR logical entity contains all the NaaS subscriber/subscription related information needed for subscription-based policies and charging control by the PCRF. This element acts as the policy solution database to store subscriber profile, quota, and state information of the Policy Server to use in its policy execution. In most cases, the SPR and the HSS are bundled in the same product.

2.4 Control and user plane separation (CUPS) of EPC nodes

CUPS was introduced as a way to overcome the following recent requirements:

  • Rich Communication Services (RCS) support: the network needs to be prepared to handle the new trend in content-rich multi-media services (e.g. OTT (Over the Top) video streaming services, Person to person video, content sharing).
  • Geographic coverage. Through the use of CUPS, NaaS operators can expand their demographic coverage in order to gain new markets, keeping centralized the signaling part of the core and distributing the user plane functionality through several regions.

CUPS in EPC nodes provides architecture enhancements for the separation of SGW and PGW functionality into SGW-C and PGW-C for Control Plane (CP), and SGW-U and PGW-U for User Plane (UP), as shown in Figure 6. This enables flexible network deployment and operation, and the independent scaling between control plane and user plane functions while not affecting the functionality of the existing nodes. The reference points added to the EPC architecture to support CUPS are:

  • Sxa: Reference point between SGW-C and SGW-U.
  • Sxb: Reference point between PGW-C and PGW-U.

Additionally, the Packet Forwarding Control Plane (PFCP) protocol was introduced for these interfaces.

Figure 6 CUPS of EPC nodes


Control and User Plane Separation allows to:

  • Reduce Latency on application service by selecting User plane nodes that are closer to the RAN.
  • Support Increase of Data Traffic by adding user plane nodes without changing the number of SGW-C and PGW-C in the network.
  • Achieve independent evolution of the CP and UP functions.

CUPS is considered an advanced feature for NaaS networks. Its implementation can be analyzed for future business cases once the network is mature, as it enables new business cases and has the potential to offer a better quality of experience, especially for applications with low latency requirements.


2.5 LTE Core Network Architecture Specification

As described above there are several options to define the EPC architecture to be deployed by the NaaS Operator. The selected option will depend on the business case, scope, and the operators core network current state. This section provides the necessary guidance to select the architecture that matches the requirements and constraints of the NaaS Operator.

  • If the NaaS operator wants to provide service to full MVNOs or MNOs and keep full control of the RAN, the minimum setup required is an MME-SGW deployment as detailed in section 2.5.1.
  • If, however, the NaaS operator also wants to provide services to light MVNOs and brand-sellers or to provide roaming interconnections, they will need to deploy a full EPC-PCC architecture. Depending on the current state of the NaaS core network, there are two options:
    • When the NaaS operator doesnt own a PS core, the deployment of the new EPC core can be done in a centralized way (see section 2.5.2 for details) or in a distributed manner using CUPS mechanism (see section 2.5.3 for details) to leverage the benefits highlighted in section 0.
    • If the NaaS operator already owns a 3G PS core, there are two more options available:
      • Integration of the new EPC elements to its core via Gn/Gp or S3/S4 interfaces. For details, see Section 2.5.4.1.
      • Upgrade of the 3G core to LTE core. For details, see Section 2.5.4.3.

2.5.1 MME-SGW deployment

If the NaaS operator scope is to provide the access network only, in order to minimize investment, but still getting full control of the RAN to serve MNOs and the full-MVNO market, the minimum deployment required is an MME SGW configuration. This deployment, as shown in Figure 7, will enable the NaaS operator to control the mobility and sessions within the E-UTRAN.

Figure 7 MME-SGW Deployment.


For this deployment option, the MME and SGW must contain different configurations (one per MNO) working independently. In this way, the interconnection to the MNOs equipment (HSS & PGW) are done as if the MME and SGW were part of the MNO network. This differs from the roaming interconnection addressed in section 3.

2.5.1.1 PCC Elements for MME-SGW deployment

No PCC elements are needed for this implementation as all policies are applied in the MNO Mobile Core network.

2.5.2 Non-existent PS core: EPC completely centralized

In the case that the NaaS operator does not have a current PS core and wants to broaden its market by providing services to MNOs and all types of MVNOs, it needs to deploy the full EPC architecture.

In this option, as shown in Figure 8, all of the EPC functions are centralized, which is the simplest way to deploy a full EPC architecture. This option is the initial recommendation for new NaaS Operators because all of its components can be implemented in one facility and matches with the EPC-in-a-box approach where all the EPC network elements are integrated as a single virtualized product.

Figure 8 EPC completely centralized


2.5.2.1 PCC Elements for Centralized EPC

This deployment, in conjunction with the PCC elements, will enable the NaaS operator to support provisioning, monitoring, control and charging of IP data services.

Please note that as indicated in section 2.2, not all the PCC elements will apply; please refer to that section for further detail.

2.5.3 Non-existent PS core: EPC with CUPS mechanism

When the NaaS operator wants to leverage the benefits that a CUPS schema brings into its new EPC deployment (refer to section 2.4 for further details), it should build an architecture as shown in Figure 9, where separation of control and user plane mechanism is used.

The plane separation at the SGW and PGW helps services which have tight latency requirements as well as overall throughput. The resulting benefits are lower network delays, higher data rates and a general improvement for the user experience. One good example for this scheme is implementing one centralized SGW-C for handling all the signaling from a country while deploying a couple of SGW-U placed where most of the served population resides (e.g. Region A & B) reducing the amount of user traffic that is carried towards the centralized EPC and the end-to-end latency, as shown in Figure 9.

Figure 9 EPC deployment with CUPS mechanism.

2.5.3.1 PCC Elements for EPC with CUPS

Compared to the completely centralized option, the PCC elements should not vary since the NaaS operator should support provisioning, monitoring, control and charging of IP data services.

Please note that as indicated in section 2.2, not all the PCC elements will apply; please refer to that section for further detail.

2.5.4 Architecture with Existent 3G Core

Whenever the NaaS Operator owns an existing 2G/3G network, it is desirable to achieve interworking of these networks with the LTE network to be deployed and this must be done at the mobile core level. Doing so, enables a fallback to 2G/3G data to allow for constant connectivity where the LTE network does not provide coverage.

There are two options to enable interworking of 2G/3G/LTE networks at the core level:

  • Option 1: By introducing new EPC nodes and interwork with the existing network through Gn/Gp or S3/S4 interfaces. See section 2.5.4.2.
  • Option 2: By fully upgrading the existing PS core to support 2G/3G/LTE. See section 2.5.4.3.

These options are analyzed in the sections below. It is important to note that before selecting one option, the NaaS Operator needs to verify compliance of the network elements with the desired mechanisms for interworking.


2.5.4.1 Option 1: Introducing new EPC nodes and interwork with the existing network through Gn/Gp or S3/S4 interfaces.

Under option 1, new EPC elements will be introduced and interconnected with the existing PS. The EPC will serve the LTE users and the existing PS will serve the 2G/3G users. 3 EPC Roaming Interconnection.

This section presents an analysis and instructions to achieve interconnection with Mobile Network Operators (MNO) sharing the RAN network of the NaaS operator. This section applies to all the deployment options discussed in Section 2 except for the MME/SGW option as explained in section 2.5.1.

Roaming is the mechanisms to interconnect to customer MNOs when implementing a Full-EPC, so it is an important aspect to consider for NaaS Operators. The available options for roaming in LTE are detailed in the sections below


3.1 Home routed

This mechanism enables MNO subscribers to access the visited network (NaaS operator network) through the home PDN gateway (H-PGW) and obtain services provided by their home networks. The subscriber's user traffic and control for their associated service is managed by the home network as shown in Figure 13.

The charging control is performed by the billing center in the home network. Since the billing center in the home network is not available to NEs in the visited network, the NEs in the visited network can implement only offline charging.

Figure 13 Home Routed Roaming Architecture.


3.2 Local breakout

When implementing local breakout, the subscriber's control for their associated service is managed by the home network, but the subscriber's user plane data is carried by the visited network (NaaS operator network) to provide access to the Internet or voice services. This scenario requires an S9 interface (H-PCRF to V-PCRF) and hence a Diameter association between the home (MNO) and the visited (NaaS) network.

In a roaming scenario there may be two PCRFs associated to the IP session of a specific UE:

  • H-PCRF that resides within the Home Public Land Mobile Network (H-PLMN). Its functions are:
    • Proxy information to the PCEF via the V-PCRF
    • The H-PCRF provides the PCC rules to the V-PCRF over the S9 reference point.

4 EPC Functional & Operational Requirements

This section presents an examination of the functional and operational characteristics of the Mobile Core, providing guidance and general criteria for NaaS Operators to establish the requirements for solution providers.


4.1 High Availability Architecture

This section presents the mechanisms, models and implementation guidelines for a resilient EPC system that complies with the desired reliability and fault tolerance to support the end user services.

In general terms, the resilience needs to be assured in two fronts: hardware level and application level.

4.1.1 Hardware Level Resilience

The hardware level resilience is achieved through the use of extra physical components working together or in backup mode to assure service continuity. This type of redundancy is the most expensive; thus, NaaS Operator should consider their budget constraints to define the level of redundancy to be implemented in their Mobile Core network. The following redundancy schemes are applied for hardware level:

  • 2N Redundancy: This is the maximum level of redundancy with a 1 on 1 configuration, where every element has a matching backup. Although this provides the higher resilience, it is cost expensive.
  • N+1 Redundancy: In this schema, there are N elements to be served by 1 backup. This model is the most cost-effective of them all, as it only keeps one backup node, so this must be seen as the minimum redundancy scheme to implement.
  • N+M Redundancy: This type of redundancy works by having N elements served by M backup elements. This model sits in between 2N and N+1 options in terms of cost and redundancy capabilities.

The previous options can be applied to internal components such as compute blades, disk drives, RAM memory modules and NICs. It will depend on the vendors hardware architecture, which redundancy mechanisms are implemented and the allowed level of customization of these mechanisms. Even a higher level of hardware redundancy is applied across different boxes following the same schemes (2N, N+M and N+1). This higher level is also the most expensive since complete server hardware will be required to take over when one of the servers fails.

As the general focus on the EPC deployment, throughout this module has been on a virtualized in-a-box EPC solution, applicable hardware resilience is divided into server or host high availability, storage high availability and network high availability, which are discussed in the following subsections.


4.1.1.1 Server or host high availability

For a server to be highly resilient, the following features need to be present as a minimum requirement for the inner components of the server:

  • Hot-swap devices: The removal and replacement of a module without powering down or shutting down the system, when a fault occurs in one of the modules such as a power supply unit, the redundant module takes over automatically without interrupting the operations. This applies to power supplies, fans, disks.
  • Hot-plug PCIe: The ability to insert or replace PCIe I/O devices through hot plugs eliminates or at least minimizes system downtime.

For higher redundancy and the associated increase in cost, complete servers are to be deployed as a backup according to the aforementioned redundancy schemes (2N, N+M, N+1).

  • 2N: Every server has a backup working in active/standby or active/active.
  • N+1: A group of servers has one backup server in case any member of the group fails. If two elements of the group fail, it is likely for it to have service outage.
  • N+M A group of N servers have M backup servers. It can occur up to M fault occurrences in the group until a service outage happens.

4.1.1.2 Storage high availability

Replication mechanisms are used in storage to achieve high availability in addition to hardware redundancy. Replication is the creation of replicas or copies of data from one storage location to another.

Minimum requirements for storage resilience are achieved by:

  • Hot-swap devices: This applies to power supplies, fans and disks on the storage device.
  • RAID: A technique based on mirroring and stripping of data that aids in recovery from disks failure. RAID is a form of N+M redundancy that utilizes redundant disk drives.
  • Replication algorithms: As an alternative to RAID, some storage solutions have specific algorithms to achieve redundancy with added features for optimization.

As a recommended redundancy schemes, although its higher cost and implementation difficulty, are the following mechanisms:

  • 2N: Every disk drive or disk array has its own backup.
  • N+1: A group of storage units has one backup server in case any member of the group fails. If two elements of the group fail, it is likely for it to have service outage.
  • N+M: Even though RAID and Replication can be seen as N+M redundancy. The hardware level of this redundancy scheme is translated on disk arrays for backup.

4.1.1.3 Network high availability

In order to achieve resilience at the hardware level in the network fabric, it is necessary to have redundant ports and networking devices (e.g. routers, switches) according to the previous schemes (2N, N+1 or N+M).

The most common and recommended scheme for EPC is 2N (ports and networking devices) since a full redundancy is key to service delivery and signaling.

Another technique that can be used in combination are bonding mechanisms. Network bonding refers to the combination of N physical network interfaces together into a single logical interface. These techniques offer redundancy and capacity aggregation. If one physical interface is down or unplugged, the others in the bonding group should be capable of handling the traffic. Additionally, the link can be over dimensioned to add one or several (1 or M) extra physical interfaces to the bonding group resulting in N+1 or N+M.


4.1.2 Application Resilience

At the application layer, meaning the software implementation of the network elements, redundancy mechanisms are used. It is vital for the NaaS operator to deploy highly resilient mechanisms to minimize service outages.

Application redundant models are based on the same principle that hardware resilience: 2N, N+1, N+M. Note that these models can apply an active/active or active/standby service model. In an active/active model, both elements are actively running the same kind of service simultaneously, this model is used for load balancing purposes, whereas in active/standby one element is the serving point while the other one is passive and ready to take over in case the active element fails. This model is typically used for failover and maintenance flexibility and simplicity.


4.1.3 Achieving a highly available EPC

In order to achieve an EPC system at the hardware level, the NaaS operator should implement all the points mentioned before in terms of redundancy and replication. If it is not possible to implement all of these at once, it is recommended to include them by parts beginning with the ones more crucial and going towards the most difficult to achieve.

At the hardware, level, Figure 15 shows the minimum resilience mechanisms to be considered by NaaS Operators:

  • Hot-swap model for server/host and storage resiliency.
  • N+1 redundancy mechanism and Hot Plug PCIe on computing resources.
  • Hot Swap and RAID technique for Storage redundancy.
  • 2N:port for connection from server infrastructure (NFVI) to L3 Switch TOR and for every external logical interface.

Figure 15 Minimum Redundancy for a virtualized Infrastructure.

  • For active/standby mode, one set of servers (NFVI) has to work in standby mode in case the active fails.
  • For active/active mode, both sets of servers (NFVI) will work in active mode, but the maximum capacity for each set must not exceed 40% of each so in case one set fails, the remaining one can take both loads (~80% capacity).

4.2.3 VoIP on LTE

As stated previously, to deploy and monetize voice services through VoIP, the NaaS Operator must provide their own VoIP application which needs to be installed in the end user devices to make use of the service. NaaS Operators This requires the application either to be pre-installed in user devices or to be downloaded and installed by the users from app stores. In addition, the native voice app in the user devices might not work; instead, the user will need to open the app to make a call.

If the NaaS Operator wants to differentiate VoIP traffic from other traffic services and deliver a better voice quality, the EPC needs to be aware of it so that this VoIP traffic must be treated differently (QoS and/or bit rate) and apply differentiated charging rates.

To optimize VoIP over an LTE network, the NaaS operator must:

  • Prioritize voice data over other data: Set up Quality of Service (QoS) rules within the PCRF for the customized voice application, ensuring that voice calls are always given the suited treatment throughout the network. The procedure is as follows:
    • The customized voice app must identify its traffic so the PGW can detect it is an IP phone call.
    • The PGW detects the traffic from the voice app and, if it is an authorized subscriber, notifies the PCRF of the occurrence.
    • The PCRF must have rules for QCI modification for the authorized users under the traffic ID. The bearer will change its QCI from 9 to 1.
    • By the PCRF command, the PGW will modify the QCI of the bearer as long as the voice call lasts. This modification will affect the SGW, eNB and UE. Additionally, the MME will be notified.
    • Once the voice call ends, the PGW will notify the PCRF and the inverse process will take place to change back to QCI9 (Best Effort).
  • Limit the number of simultaneous calls to avoid exhausting the capacity of the network and to minimize connections that will be blocked in case of insufficient resources, due to QoS guarantees. This limitation can be performed by the PCRF because it will have the responsibility of making the QoS changes and, consequently, to limit them by predefined rules.
    • The PGW would have a counter for simultaneous voice call control so when this limit defined by the NaaS Operator is reached, it can deny more QCI modifications.
  • Implement a VoIP Server: The entity for handling the voice over IP calls from all UE in the NaaS Operators network. This server has the capability of routing the calls from outside networks and between subscribers.

In conclusion, to implement VoIP over LTE, the NaaS operator needs to add an element to its EPC deployment (VoIP server) additional to the PCC network elements deployed, and apply the necessary configuration to assure the QCI level for voice traffic on their network.


4.2.4 VoLTE

Voice calls over LTE are recognized as the industry-agreed progression of voice services across mobile networks deploying LTE radio access technology. The implementation of VoLTE offers many benefits to the NaaS operator. Some of these benefits are:

  • One RAN technology can support both high quality voice and best effort data.
  • Efficient use of spectrum, as LTE is aware of the requirements that voice packets need.
  • Can be deployed in parallel with video calls over LTE and RCS multimedia services, including video share, multimedia messaging, chat and file transfer.
  • Delivers an unusually crisp calling experience.
  • Provides rapid call establishment time.

VoLTE deployment has also some drawbacks such as:

  • The requirement for an IMS core. If the NaaS Operator does not have an agreement with an IMS owner (e.g. one of the customer MNOs sharing the RAN), they must deploy an IMS core with its EPC core to offer VoLTE.
  • Many UEs (specially those refurbished) wont support VoLTE.
  • The provisioning can be challenging since it requires a more precise subscriber profile.
  • The network signaling has tight delay requirements and complex configuration for roaming scenarios, making more difficult the initial setup of the service.

VoLTE utilizes the IMS platform to provide the service layer for providing Multimedia Telephony. It defines an architecture for the convergence of audio, video and data and enables integrated multimedia sessions over fixed and wireless networks.

If NaaS Operators opt for an MME/SGW deployment, they must configure on the RAN, MME and SGW the corresponding parameters to enable the VoLTE traffic and signaling. This will allow the other MNOs to offer VoLTE and generate better revenues for both parties.

If the NaaS Operator wants to offer VoLTE services by themselves, it will be needed to implement a full EPC along with the PCC nodes.

The overall VoLTE architecture including EPC and IMS is shown in Figure 17.

Figure 17 VoLTE architecture

A VoLTE call involves all the elements on the EPC, PCC, and IMS platforms:

  • EPC: Responsible for the call signaling transport, including call signaling in SIP form and voice packets in the form of RTP. Based on these 2 different flows and their requirements, the EPC nodes set 2 dedicated bearers for its transportation.
  • PCC: Responsible for resource reservation for the dedicated bearers in order to fulfill the requirements for call signaling and voice packets. Additionally, once the call is set up, the PCC will ensure that the call is billed appropriately.
  • IMS: Responsible for users registration and call control.
    • Registration: Before making or receiving calls, VoLTE subscribers need to register to the IMS. This will assure the authentication of the user and the IMS network.
    • Call control: When a call is set, the UEs involved will send signaling information to ensure the call establishment, continuity and termination.

In summary, to implement VoLTE, the NaaS operator needs to fully deploy the IMS Core if no IMS operators are available in the zone. The PCC elements must also have to be integrated into the EPC for billing and dedicated bearer establishment. It is mandatory for the EPC elements to be capable of supporting VoLTE features (extra cost for licensing may apply). Additionally, UE must be VoLTE-ready in order to take advantage of it.

Finally, it must be noted that although VoLTE is the standard in the industry to offer voice services in LTE, it is complex to set up and the NaaS Operators must perform a self-assessment of their technical and financial capabilities before deciding to implement VoLTE.


5 Telco Cloud Architecture

Network Functions Virtualization (NFV) is a way to virtualize network services, such as routers, firewalls, load balancers and EPC elements, that have traditionally been run on proprietary hardware. In a virtualized environment these services are packaged as virtual machines (VMs) on commodity hardware, which allows service providers to run their network on standard servers instead of proprietary hardware.

The Network Function Virtualization Infrastructures (NFVI) physical resources (e.g. CPUs, RAM, Storage) are virtualized by the Virtual Infrastructure Manager (VIM). Management and Orchestration (MANO) manages all the virtualized resources to control the lifecycle of the Virtual Network Functions (VNFs) through the VNF Manager (VNFM). The VNF Manager (VNFM) uses the virtualized resources to create VNFs.

The VNFs are created according to the functions to be supported (e.g. Control Plane, User Plane, Routing) and then mapped into Network Elements (NE). These NE can be an MME, SGW, PGW, etc. All of the above elements are depicted in Figure 18.

Figure 18 Graphic representation of the NFV Solution for a vEPC

The mapping between the VNFs and NEs can be 1:1, 1:N or N:1 [VNF:NE] and represents the complexity of the VNF. See Figure 19.

  • One VNFs can support multiple NEs (1:N) so that the complete vEPC can be mounted on one VNF.
  • One NE can be mapped into several VNFs (N:1). Each VNFs can be assigned to routing, Control Plane or User Plane.
  • One VNF can support the complete functionality of one NE (1:1). In this scenario, the number of Mobile Core Elements are the number of VNF the solution must have.

Figure 19 Simplified representation of VNF to NE mapping.

With NFV, it is not necessary to have dedicated hardware for each network function. NFV improves scalability and agility by allowing service providers to deliver new network services and applications on-demand, without requiring additional hardware resources. In the following sections, it is presented the NFV overview, architecture definition process and recommendations.


5.1 NFV & Virtualization Technologies Overview

The evolution of LTE towards a virtualized platform NFV-based is guided by the following benefits that this technology brings:

  • OPEX reduction: Highly adaptive network scaling means that utilization of the network is optimized, due to the ability to scale the network up by adding more nodes as an automated process.
  • CAPEX reduction: Industry-standard servers form the hardware of the system with shared computing and storage for applications. This means that there are significant cost savings since there is no need for proprietary hardware.
  • Scalability and Flexibility: Based on the network conditions, network capacity can be scaled up or down much more easily. The Virtual Network Function (VNF) components that support a given service can be scaled up or down based on load conditions, e. g. traffic load.
  • New Revenue Opportunities: New applications can be much more easily brought to market using a standardized NFV architecture. In a traditional network, the deployment of a new Network Element (NE) would take several weeks; in a virtualized environment, this could be reduced to hours or even minutes.

5.1.1 NFV Functional Blocks

The Functional Blocks of the NFV framework are described in the following subsections.

5.1.1.1 Network Function Virtualization Infrastructure (NFVI)

The NFVI is the totality of all hardware and software components which build up the environment in which VNFs are deployed, managed and executed. The physical hardware resources include computing, storage and networking elements that provide processing, storage and connectivity to VNFs through the virtualization layer (e.g. hypervisor).

The virtualization layer is responsible for:

  • Abstracting and logically partitioning physical resources, commonly as a hardware abstraction layer.
  • Enabling the software that implements the VNF to use the underlying virtualized infrastructure.
  • Ensures VNFs are decoupled from hardware resources and therefore, the software can be deployed on different physical hardware resources.

Examples of NFVI are physical servers, physical processors, disk drives, NICs and networking elements (e.g. switches/routers) in combination with the virtualization layer.

5.1.1.2 Management and Orchestration (MANO)

The NFV MANO (Management and Orchestration) is a framework for the management and orchestration of all resources in a virtualized data center including computing, networking, storage, and virtual machine (VM) resources. The main focus of NFV MANO is to allow flexible on-boarding and lifecycle management of NFVs, avoiding the chaos that can be associated with rapid spin-up of network components.

It is divided into the Network Function Virtualization Orchestrator (NFVO), VNF Manager (VNFM) and the Service, VNF and Infrastructure Description and Virtual Infrastructure Manager (VIM). All of these components are discussed in the following paragraphs.

Virtualized Infrastructure Manager (VIM)

The VIM is in charge of NFVI resource management, which conveys:

  • Inventory of software (e.g. hypervisors), computing, storage and network resources dedicated to NFVI
  • Management of infrastructure resource and allocation, e.g. increase resources to VMs, improve energy efficiency, and resource reclamation.

VIM performs operations for:

  • Visibility into and management of the NFV infrastructure.
  • Root cause analysis of performance issues from the NFV infrastructure perspective.
  • Collection of infrastructure fault information.
  • Collection of information for capacity planning, monitoring, and optimization.

Examples of VIMs are OpenStack, OpenVIM, VMware vSphere.

Network Function Virtualization Orchestrator (NFVO)

The NFV Orchestrator is in charge of the orchestration and management of NFV infrastructure and software resources, and realizing network services on NFVI.

Examples of NFVO are Proprietary NFVO, OpenMANO, OpenSource MANO and ONAP.

VNF Manager (VNFM)

Responsible for VNF lifecycle management (e.g. instantiation, update, query, scaling, termination).

Examples of VNFM are Proprietary VNFM and Tacker.

Service, VNF and Infrastructure Description

Templates/descriptors used internally for management and orchestration.

5.1.1.3 Virtual Network Functions (VNF)

A VNF is a virtualized version of a network function, capable of running on an NFV Infrastructure (NFVI) and being orchestrated by an NFV Orchestrator (NFVO) and VNF Manager.

For the virtual EPC, the basic case is when each element (e.g. MME, SGW, PGW) is implemented by one VNF. This is known as one-to-one mapping. In addition, other mappings can be deployed, for example a single NE comprised of various VNFs or a VNF containing several NEs.

Element Management (EM)

Within the VNF functional block it is also considered the Element Management System (EMS) which performs the typical management functionality for one or several VNFs. Similar to the EMS for legacy systems, this is a software application and it can be instantiated in the same infrastructure than the VNF itself.

5.1.2 Virtualization technologies

NFV relies on server virtualization technology, these are divided on:

  • Hypervisor-based solutions: A hypervisor is a software program that partitions the resources of a single hardware host and creates Virtual Machines (VM) isolated from each other with a different operating system instance. Each virtual machine appears to have the host's processor, memory and other resources, all to itself. The operations and procedures between NFVI and VIM were designed to act upon virtual machines rather than virtualization containers in general. Some of the benefits of using Hypervisor -based solution are:
    • More efficient use of hardware. Being able to virtualize servers means being able to run more operating systems on your hardware. This leads to efficiencies in everything from maintenance to energy costs and rack space to cooling.
    • Easier and faster management & provisioning. It is much easier and faster to spin up and down virtual servers than provision new hardware. Creating backups and server snapshots is much simpler as well.
    • Improved disaster recovery. Because it is easy to create backups and recover to snapshot images, disaster recovery is easier and faster.
    • Simplified testing. Testing in production isnt a great idea. Dedicating a server to testing isnt too efficient if testing is done for many different operating systems. With VMs, each virtual server has its own logically isolated operating system providing flexibility.
  • OS Containers: Container-based virtualization, also called operating system (OS)-level virtualization, is an approach to virtualization which allows multiple isolated user space instances on top of a kernel space within the same OS. The isolated guests are called containers.
    • Containers are lightweight. Containers use only the binaries and libraries they need. This means they boot significantly faster than VMs. Being more lightweight also means less hardware and fewer operating systems are needed.
    • Containers enable microservices. In short, microservices are small services that communicate with one another to create a larger application or service. By decoupling individual microservices testing becomes easier, single points of failure are reduced, and development velocity is increased.

As a summary, the most used virtualization technique is the hypervisor-based one, due to its maturity, however, the future points out to container-based platforms to leverage their use through microservices-based and DevOps optimized architectures. As a starting point, NaaS operator should choose their NFV platform on hypervisor-based technologies. Once their platform is mature, they can adopt a hybrid approach, container and hypervisor-based, to leverage the benefits that both technologies bring. However, it is important to note that the NaaS Operator must consult with the potential vendors what is the recommended/supported virtualization solution for their product.

5.2.2 OSS/BSS Integration with NFV

The Operations Support Systems and Business Support Systems (OSS/BSS) provide network and customer data for back office activities.

The OSS systems support internal network requirements such as network planning, maintaining network inventory, provisioning services, configuring network components and managing.

On the other hand, the BSS provides the capability for dealing with customers, supporting processes such as taking orders, processing bills, and collecting payments.

The OSS/BSS ecosystem integration will enable the NaaS operator to provide customization, management of product content, building profiles of their customers, and revenue and billing management, or even data analytics. Although it is not strictly mandatory to implement OSS/BSS systems, it is strongly recommended to take advantage of these features. In addition, a more complete, unified and automated platform is achieved.

The NFV platform enables throughout the northbound interfaces the capability to integrate these entities into the new virtualized paradigm. However, in order to get the benefits of this unification, the NaaS operator should demand the following requirements to the OSS/BSS and NFV vendor:

  • If possible, a NaaS operator should always opt for an OSS/BSS to take advantage of all the benefits, agility and speed NFV can give.
  • The OSS/BSS platform needs to support flexible service modeling (e.g. adopt IETF modeling approach).
  • The OSS/BSS must support the communication between the traditional OSS/BSS and the MANO framework according to what is defined in ETSI SOL 005, RESTful Application Programming Interface (APIs), and to ETSI IFA 013.

6 Architecture Module Recommendation

This section provides a methodology to consolidate the decisions taken by the NaaS Operator to define the Mobile Core Architecture and generate a final Architecture Specification.


6.1 Architecture Specification Format & Structure

After going through this module, the NaaS Operator will be able to generate the main deliverable which is the Mobile Core Architecture Specification. This specification shall contain the overall technical solution, basic design rules, technologies and concepts required to describe Mobile Core Network. The Architecture Specification shall contain the following aspects:

  • Overview: High-level description that summarizes the Mobile Core Architecture.
  • EPC & PCC Architecture Requirements: According to section 2, it must be defined a list of all the elements of the EPC & PCC, indicating their main functions and responsibilities. If any special function is to be supported, it must be specified. It also must contain the interfaces needed.
  • EPC Roaming Interconnection: Corresponding to section 3, it must be tailored a strategy to cover all the interconnection to partner/customer MNOs.
  • Redundancy Requirements: Based on section 4, it must be specified what will be the redundancy mechanisms.
  • Voice Strategy: According to section 4.2, if a voice technology is to be implemented, it has to be selected and define its integration to the architecture.
  • NFV Architecture: Based on section 5, the components and specifics of the NFV technology must be identified and registered into the virtualized architecture. Additionally, it must be decided if an OSS/BSS system is to be integrated.


6.2 Architecture Recommendation Generation

The architecture design is based on the several and unique inputs of the NaaS Operator. A Mobile Core Wizard was developed in order to ease this design process to the NaaS Operator. Further customizations can be applied to the output of this tool.

The final Architecture Recommendation must be generated following the format and the structure established. NaaS Operator can use the Architecture Template Document as a reference to create their own version.
